## 新增需求

### 需求:索赔事项实体定义

系统必须创建 LawsuitClaim 实体，包含以下字段：id（主键）、taskName（任务名称）、taskCode（任务编码）、taskType（任务类型）、courtDocumentNo（法院文书号码）、isMajorCase（是否重大案件）、isClaimPaymentDispute（是否理赔给付纠纷）、isRefundDispute（是否退保纠纷）、occurrencePeriod（发生区间）、claimedOrgProvince（被索赔机构-省）、claimedOrgCity（被索赔机构-市）、claimedOrgDistrict（被索赔机构-区/县）、claimedOrgOther（被索赔机构-其他）、claimOrg（索赔机构）、factsAndReasons（事实与理由）、prosecutionProbability（起诉可能性评估）、loseProbability（败诉可能性评估）、attachments（附件路径）、approvalStatus（审批状态）、createdBy（创建人）、createdAt（创建时间）、updatedAt（更新时间）。

#### 场景:实体持久化

- **当** 系统保存一条新的索赔事项
- **那么** 所有字段必须正确持久化到 lawsuit_claim 数据库表中

### 需求:任务编码自动生成

系统必须在创建索赔事项时自动生成任务编码，格式为"诉讼[四位年份]第[五位流水号]号"。

#### 场景:生成第一条编码

- **当** 当前年份为2026年，数据库中无2026年的记录
- **那么** 系统必须生成编码"诉讼[2026]第00001号"

#### 场景:生成后续编码

- **当** 当前年份为2026年，数据库中2026年最大流水号为00005
- **那么** 系统必须生成编码"诉讼[2026]第00006号"

#### 场景:跨年重置流水号

- **当** 当前年份为2027年，数据库中无2027年的记录
- **那么** 系统必须生成编码"诉讼[2027]第00001号"

### 需求:创建索赔事项接口

系统必须提供 POST /api/lawsuit-claims 接口，用于创建新的索赔事项。

#### 场景:成功创建

- **当** 客户端发送包含所有必填字段的 POST 请求
- **那么** 系统必须创建索赔事项，自动生成任务编码，设置状态为 DRAFT，返回 201 状态码及创建的数据

#### 场景:必填字段缺失

- **当** 客户端发送缺少必填字段的 POST 请求
- **那么** 系统必须返回 400 状态码及字段验证错误信息

### 需求:查询索赔事项列表接口

系统必须提供 GET /api/lawsuit-claims 接口，支持分页查询和条件筛选。

#### 场景:分页查询

- **当** 客户端发送 GET 请求，携带 page=1&size=10
- **那么** 系统必须返回第1页、每页10条的索赔事项列表及总数

#### 场景:按审批状态筛选

- **当** 客户端发送 GET 请求，携带 approvalStatus=DRAFT
- **那么** 系统必须仅返回审批状态为草稿的索赔事项

### 需求:查询索赔事项详情接口

系统必须提供 GET /api/lawsuit-claims/{id} 接口，返回单条索赔事项的完整信息。

#### 场景:查询存在的记录

- **当** 客户端请求一条存在的索赔事项
- **那么** 系统必须返回该事项的完整信息

#### 场景:查询不存在的记录

- **当** 客户端请求一条不存在的索赔事项
- **那么** 系统必须返回 404 状态码

### 需求:更新索赔事项接口

系统必须提供 PUT /api/lawsuit-claims/{id} 接口，用于更新索赔事项。仅草稿和已驳回状态的事项允许编辑。

#### 场景:更新草稿状态的事项

- **当** 客户端对草稿状态的事项发送 PUT 请求
- **那么** 系统必须更新对应字段并返回更新后的数据

#### 场景:更新非草稿非驳回状态的事项

- **当** 客户端对非草稿非驳回状态的事项发送 PUT 请求
- **那么** 系统必须返回 400 状态码，提示"当前状态不允许编辑"

### 需求:审批操作接口

系统必须提供 POST /api/lawsuit-claims/{id}/approve 接口，用于执行审批操作（提交、通过、驳回）。

#### 场景:提交审批

- **当** 客户端发送 action=SUBMIT 的审批请求，事项状态为 DRAFT 或 REJECTED
- **那么** 系统必须将状态更新为 PENDING_OFFICE_REVIEW 并记录审批日志

#### 场景:通过审批

- **当** 客户端发送 action=APPROVE 的审批请求
- **那么** 系统必须根据当前状态将事项推进到下一审批阶段并记录日志

#### 场景:驳回审批

- **当** 客户端发送 action=REJECT 的审批请求，包含驳回原因
- **那么** 系统必须将状态更新为 REJECTED 并记录审批日志

### 需求:文件上传接口

系统必须提供 POST /api/lawsuit-claims/upload 接口，用于上传附件文件。

#### 场景:成功上传文件

- **当** 客户端上传一个合法文件
- **那么** 系统必须将文件保存到服务器并返回文件路径

#### 场景:文件过大

- **当** 客户端上传的文件超过大小限制
- **那么** 系统必须返回 400 状态码，提示文件过大

### 需求:审批日志查询接口

系统必须提供 GET /api/lawsuit-claims/{id}/approval-logs 接口，返回指定事项的审批操作日志。

#### 场景:查询审批日志

- **当** 客户端请求某条索赔事项的审批日志
- **那么** 系统必须按时间倒序返回所有审批记录，包含操作人、操作时间、操作类型、审批意见
