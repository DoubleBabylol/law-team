## 上下文

本项目是一个权限管理系统（PMS），技术栈为：
- **后端**：Spring Boot 3.4 + JPA + SQLite + JWT 认证 + Lombok
- **前端**：Vue 3 + TypeScript + Element Plus + Pinia + Vue Router + Vite
- **数据库**：SQLite（单文件数据库）

现有系统已包含用户管理、角色管理、菜单管理、权限管理等基础模块。本次需新增"重大索赔事项"业务模块，是诉讼管理的第一层业务模型。

## 目标 / 非目标

**目标：**
- 实现重大索赔事项信息录入的完整前后端功能
- 实现两级审批流程（总经办 → 总审核）
- 任务编码按规则自动生成
- 省市区三级联动选择
- 文件附件上传功能

**非目标：**
- 不实现复杂的工作流引擎，使用简单状态机实现审批
- 不实现诉讼后续阶段的业务流程（本次仅第一层）
- 不实现通知提醒功能（如审批通知）
- 不对接外部省市区数据接口，使用内置静态数据

## 决策

### 1. 审批流程方案：简单状态机 vs 工作流引擎

**选择：简单状态机**

理由：当前审批链路固定（草稿 → 待总经办审批 → 待总审核 → 已通过/已驳回），无需引入 Activiti/Flowable 等重量级工作流引擎。使用枚举状态字段 + 审批记录表即可满足需求，降低复杂度。

审批状态流转：
- `DRAFT`（草稿）→ 提交 → `PENDING_OFFICE_REVIEW`（待总经办审批）
- `PENDING_OFFICE_REVIEW` → 通过 → `PENDING_FINAL_REVIEW`（待总审核）
- `PENDING_OFFICE_REVIEW` → 驳回 → `REJECTED`（已驳回）
- `PENDING_FINAL_REVIEW` → 通过 → `APPROVED`（已通过）
- `PENDING_FINAL_REVIEW` → 驳回 → `REJECTED`（已驳回）
- `REJECTED` → 重新提交 → `PENDING_OFFICE_REVIEW`

### 2. 省市区数据方案：静态 JSON vs 后端接口

**选择：前端静态 JSON 数据**

理由：省市区数据相对固定，直接在前端维护一份中国行政区划 JSON 即可，无需后端接口。减少网络请求，提升选择体验。

### 3. 文件上传方案：本地存储

**选择：服务器本地文件存储**

理由：当前为 SQLite 小型项目，不适合对接 OSS。文件存储在服务器本地 `uploads/` 目录，数据库仅存储文件路径。后续可扩展为 OSS。

### 4. 任务编码生成方案

**选择：后端服务端生成**

使用数据库查询当前年份最大流水号 +1 的方式生成。格式：`诉讼[yyyy]第[00000]号`。在 Service 层实现，确保并发安全（使用 `synchronized` 或数据库锁）。

### 5. 数据库表设计

新增两张表：
- `lawsuit_claim`：重大索赔事项主表，包含所有业务字段及审批状态
- `lawsuit_claim_approval_log`：审批记录表，记录每次审批操作

## 风险 / 权衡

- **[SQLite 并发限制]** → SQLite 写锁粒度较粗，高并发场景可能受限。缓解：当前系统用户量不大，短期可接受；长期可考虑迁移到 MySQL/PostgreSQL。
- **[任务编码并发冲突]** → 多用户同时创建可能导致编码重复。缓解：使用 synchronized 关键字或数据库唯一约束保证唯一性。
- **[省市区数据维护]** → 静态数据可能过时。缓解：行政区划变更频率低，定期更新即可。
- **[文件上传安全]** → 需校验文件类型和大小。缓解：后端增加文件类型白名单和大小限制。
